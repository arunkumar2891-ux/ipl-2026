Key Features ť Technical Deep Dive

Masked OTP Authentication

6-digit OTP rendered via password-type inputs with auto-focus progression ť each digit auto-advances to the next field. Auto-submits the moment the 6th digit is entered (useEffect watching OTP length). Full clipboard paste support that distributes digits across all 6 fields. Mobile-optimized with inputMode="numeric" for the number keyboard. OTP validated server-side against a Supabase table; session stored in localStorage.

Live Countdown Timers

Two timer systems: a per-match countdown computing the delta between now and the match's dateutc minus 15 minutes (the cutoff), and a global next-match countdown derived from the upcoming fixtures API. Match schedule is fetched from the Supabase fixtures table ť the single source of truth ť via React Query with 5-minute auto-refetch. The prediction form disables itself reactively when the countdown hits zero ť and the backend independently verifies the cutoff by querying the fixtures table before accepting any prediction.

Pari-Mutuel Points Engine

Server-side scoring pipeline in a single Express route: fetches all rows from final_prediction for the match number, groups by bgroup, applies per-user multipliers (configurable map ť 1x default, 2x and 5x for specific users), counts weighted winners vs losers, computes matchWinAmount = (losersCount ť bidValue) / winnersCount, then writes per-player rows to matchdata. Cumulative leaderboard is built by fetching the previous match's leaderboard and adding the new matchWinAmount. Playoff escalation: base 20 ? 50 for qualifiers ? 100 for the final. "No Result" handling (rain/abandonment) carries forward the previous leaderboard with matchwinamount: 0 and deletes stale rows before re-inserting.

Last 5 Form Guide

Dedicated GET /api/leaderboard/form endpoint queries the leaderboard table, extracts the 5 most recent distinct match numbers via Set + sort + slice, fetches matchwinamount for each user across those matches, and classifies: positive = "W", zero = "NR" (no result), negative = "L". Frontend renders three dot colors ť glowing green for wins, orange for no-result, red for losses (CSS box-shadow with rgba glow) ť with hover tooltips showing "Match N: Won/Lost/No Result". The sliding window advances automatically ť zero manual config needed.

Multi-Group Prediction

Users belonging to multiple groups (e.g., G1 and G2) have their groups resolved from a members array at submit time. The frontend fires Promise.all() across all groups, inserting one prediction row per group per submission. Each group has its own independent leaderboard and pari-mutuel pool. The admin's "generate unbids" RPC also respects group membership.

Admin Console

Email-whitelisted via a Supabase admins table check (GET /api/admin/checkAdmin). Fetches today's match numbers and team names via RPC get_todaymatches. "Generate Unbids" calls RPC insert_unbid_predictions to backfill default entries for users who didn't predict. "Declare Winner" triggers the full scoring pipeline ť matchdata write + leaderboard update ť in one click. The admin can also switch the active user email for testing.

Active User Highlighting

The logged-in user's row is highlighted across both the leaderboard and bid details tables. The active email is read from localStorage, matched against the members array to resolve name(s), and a Set lookup applies bg-primary/10 + font-semibold to matching rows ť so users can instantly spot themselves in the standings and prediction logs.

Dark-Mode UI with Animations

Ship-ready dark theme via Tailwind CSS custom properties (CSS variables for background, card, border, muted, accent). shadcn/ui components built on Radix UI headless primitives for accessibility. Framer Motion powers tab transitions (AnimatePresence + layoutId for the active tab indicator), row fade-ins on the leaderboard, and page-level mount animations. Gold/silver/bronze border accents on the top 3 leaderboard rows.

Data Layer & Caching

TanStack React Query v5 manages all server state with declarative queryKey-based caching. Leaderboard and form guide queries run in parallel on mount. Today's matches and upcoming fixtures are fetched via dedicated API endpoints with 5-minute auto-refetch intervals. After a prediction submission, queryClient.refetchQueries invalidates the bids cache so the prediction log updates instantly. The backend uses Supabase JS client with .select(), .eq(), .in(), .order(), .maybeSingle() for type-safe query building, and .rpc() for stored procedures. API inputs are validated and sanitized on both ends ť email params are encodeURIComponent'd on the client and trimmed on the server, with proper Array.isArray() safety checks on all RPC responses.

Fixtures as Single Source of Truth

Match schedule migrated from a hardcoded frontend TypeScript module to a Supabase fixtures table ť the single source of truth for all match data (matchnumber, dateutc, home, away, location). The backend prediction endpoint now fetches the match start time from the fixtures table rather than trusting client-supplied matchStartUtc, making cutoff enforcement fully server-authoritative. Two new endpoints (GET /api/fixtures/today and GET /api/fixtures/upcoming) serve the frontend with date-range filtered queries, eliminating stale hardcoded data.
