Key Features — Technical Deep Dive

OTP Auth + JWT Tokens

6-digit OTP rendered via password-type inputs with auto-focus progression and auto-submit on the 6th digit. Clipboard paste distributes digits across all fields. Mobile-optimized with inputMode="numeric". OTP validated server-side against Supabase with a strict rate limit (5 attempts per 15 min). On success, the backend issues a short-lived JWT (jsonwebtoken, 15-min expiry) returned to the frontend and stored in localStorage. All protected routes (prediction, admin) require the JWT via Authorization: Bearer header. The backend extracts the user email from the token — no email in the request body — so identity is server-verified, not client-supplied.

Live Countdown Timers

Two timer systems: a per-match countdown computing the delta between now and the match's dateutc minus 15 minutes (the cutoff), and a global next-match countdown derived from the upcoming fixtures API. Match schedule is fetched from the Supabase fixtures table — the single source of truth — via React Query with 5-minute auto-refetch. The prediction form disables itself reactively when the countdown hits zero — and the backend independently verifies the cutoff by querying the fixtures table before accepting any prediction.

Pari-Mutuel Points Engine

Scoring logic extracted into a shared calculateMatchResult service (used by both the admin API and the auto-result cron job): fetches all rows from final_prediction for the match number, groups by bgroup, applies per-user multipliers (configurable map — 1x default, 2x and 5x for specific users), counts weighted winners vs losers, computes matchWinAmount = (losersCount × bidValue) / winnersCount, then writes per-player rows to matchdata. Cumulative leaderboard is built by fetching the previous match's leaderboard and adding the new matchWinAmount. Playoff escalation: base 20 → 50 for qualifiers → 100 for the final. "No Result" handling (rain/abandonment) carries forward the previous leaderboard with matchwinamount: 0 and deletes stale rows before re-inserting.

Last 5 Form Guide

Dedicated GET /api/leaderboard/form endpoint queries the leaderboard table, extracts the 5 most recent distinct match numbers via Set + sort + slice, fetches matchwinamount for each user across those matches, and classifies: positive = "W", zero = "NR" (no result), negative = "L". Frontend renders three dot colors — glowing green for wins, orange for no-result, red for losses (CSS box-shadow with rgba glow) — with hover tooltips showing "Match N: Won/Lost/No Result". The sliding window advances automatically — zero manual config needed.

Multi-Group Prediction

Users belonging to multiple groups (e.g., G1 and G2) have their groups resolved from the members array at submit time via a Set-based dedup. The frontend fires Promise.all() across all groups, inserting one prediction row per group per submission. Each group has its own independent leaderboard and pari-mutuel pool. The admin's "generate unbids" RPC also respects group membership. A custom useLoggedInUser hook reads the logged-in email from localStorage and maps it to detected groups, which the leaderboard and bid details tables use to filter views per group. Bid details include both raw per-player prediction logs and aggregated per-team analytics per match/group.

Admin Console

Double-gated: JWT authentication + Supabase admins table check via requireAdmin middleware. Admin endpoints switched from GET to POST with JSON bodies to prevent CSRF. Fetches today's match numbers via RPC get_todaymatches. "Generate Unbids" marks the match as started and calls RPC insert_unbid_predictions. "Declare Winner" calls the shared calculateMatchResult service. Dedicated admin list endpoint provides admin names for the login dropdown. Admin-specific rate limit: 20 requests per 15 minutes. Most result processing is now fully automated by the background cron, but the admin console remains as a manual override and fallback.

Rate Limiting & Security

Three-tier express-rate-limit configuration: general API (25 req/1 min), OTP endpoint (5 req/15 min to prevent brute-force), and admin operations (20 req/15 min). Request body size capped at 10kb via express.json({ limit }). CORS locked to the production domain. All admin endpoints require both JWT auth and admin role verification (checked against a Supabase admins table, not just an email whitelist). Prediction endpoint extracts email from JWT — not from the request body — eliminating identity spoofing.

Active User Highlighting

The logged-in user's row is highlighted across both the leaderboard and bid details tables. A custom useLoggedInUser hook reads the email from localStorage, maps it against the members array to resolve name(s) and group memberships, and a Set lookup applies bg-primary/10 + font-semibold to matching rows — so users can instantly spot themselves in the standings and prediction logs.

Dark-Mode UI with Animations

Ship-ready dark theme via Tailwind CSS custom properties (CSS variables for background, card, border, muted, accent). shadcn/ui components built on Radix UI headless primitives for accessibility. Lucide React provides crisp SVG icons throughout the UI (Trophy, Target, BarChart3, Shield, ArrowUpDown, Calendar, etc.). Framer Motion powers tab transitions (AnimatePresence + layoutId for the active tab indicator), row fade-ins on the leaderboard, and page-level mount animations. Gold/silver/bronze border accents on the top 3 leaderboard rows.

Data Layer & Caching

TanStack React Query v5 manages all server state with declarative queryKey-based caching. Leaderboard and form guide queries run in parallel on mount. Today's fixtures are fetched with staleTime: 0 and refetchOnMount: "always" to ensure fresh data, plus a 5-minute auto-refetch interval. After a prediction submission, queryClient.refetchQueries invalidates the bids cache so the prediction log updates instantly. The backend uses Supabase JS client with .select(), .eq(), .in(), .order(), .maybeSingle() for type-safe query building, and .rpc() for stored procedures. API inputs are validated and sanitized on both ends — email params are encodeURIComponent'd on the client and trimmed on the server, with proper Array.isArray() safety checks on all RPC responses.

Fixtures DB, Auto-Postpone & Auto-Result

Match schedule stored in a Supabase fixtures table with matchstarted and resultprocessed flag columns. The backend prediction endpoint fetches match start time from fixtures rather than trusting client input, making cutoff enforcement fully server-authoritative. Two node-cron background jobs run in matchChecker.js: (1) Match-start checker runs every 10 minutes between 9:00–23:59 UTC, queries today's unstarted fixtures, cross-checks against the live CricAPI. If found, marks the fixture as started and auto-generates unbids. If NOT found within the check window, auto-pushes dateutc forward by 25 minutes for rain delays. (2) Result checker runs every 30 minutes between 13:00–23:59 UTC with a dynamic time gate (double-header days start at 7:00 PM IST, single-match days at 11:30 PM IST). It polls CricAPI, parses the winner from the match status string (handling "won by", super overs, no results, and abandoned matches), maps the winning team's full name to its shortname via a teams DB lookup, then calls the shared calculateMatchResult service to auto-trigger scoring and leaderboard generation. A retry mechanism re-checks every 30 minutes for pending results. The resultprocessed flag prevents duplicate processing.
